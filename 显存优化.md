> # 长序列训练 MoE Block 冲第一实现方案
>
> ## Summary
>
> 目标是在保持 `MoEBlockOptimized` 与 baseline 数学等价、`state_dict` 权重名完全一致的前提下，重点压低 8K 和 128K 训练峰值显存，并把速度损失控制在可接受范围。核心路线采用“PyTorch 自定义 autograd + 分块重计算 + 去除 one_hot 路由临时张量”，先保证正确性，再在 H20 上调 chunk size 和热点实现。
>
> 开发策略采用 Mac mini 本地写代码、做小规模 CPU correctness smoke test；租 H20 只做 CUDA 正确性、显存、速度和参数调优，减少租卡时间浪费。
>
> ## Key Implementation
>
> - 保持提交入口为 `solution.py` 中的 `MoEBlockOptimized(config)`，参数结构严格沿用模板：`gate.weight`、`experts.gate_up_proj`、`experts.down_proj`、`shared_expert.*.weight`、`post_norm.weight` 不改名、不换含义。
>
> - Router 部分先使用 PyTorch 原生实现：`linear -> softmax(dtype=float32) -> topk -> topk 权重归一化 -> cast`。这部分显存相对可控，且保留原生 autograd 能降低 gate 梯度出错风险。
>
> - Routed experts 去掉 baseline 的 `F.one_hot(top_k_index, num_classes=128)`。改为把 `[N, top_k]` 的 expert index 展平成 `[N * top_k]`，按 expert 排序，得到 `sorted_expert_ids`、`sorted_token_idx`、`sorted_top_pos`、`expert_counts`，逐 expert 处理 token。
>
> - Routed experts 使用自定义 `torch.autograd.Function`。forward 在 `no_grad` 下逐 expert 计算 `gate/up -> SiLU(gate) * up -> down -> routing_weight 加权 -> index_add`，只保存输入、路由索引、路由权重和参数引用，不保存每个 expert 的中间激活。
>
> - Shared expert 使用分块重计算。forward 按 token chunk 计算 `down(SiLU(gate_proj(x)) * up_proj(x))` 并累加到 combined buffer；backward 按相同 chunk 重算 shared expert，累加 `x`、`gate_proj.weight`、`up_proj.weight`、`down_proj.weight` 梯度。
>
> - MoE core 和 RMSNorm 合并到一个自定义 autograd Function 内部：forward 先构造 `combined = routed_output + shared_output`，再分块执行 RMSNorm 输出；不把 `routed_output`、`shared_output`、`combined` 都作为 autograd 图里的长期激活保存。
>
> - Backward 中先重算 `combined`，用分块 autograd 或手写公式得到 RMSNorm 对 `combined` 和 `post_norm.weight` 的梯度，再分别把 `grad_combined` 喂给 routed experts 和 shared expert 的分块重算逻辑。
>
> - 默认 chunk size 先设为 `8192` tokens；H20 调参时测试 `2048 / 4096 / 8192 / 16384`。最终写成内部常量或根据 `N` 自动选择，不依赖评测方额外传环境变量。

> # 长序列 MoE Block 冲第一实现方案
>
> ## Summary
>
> 结合 `moe-training` 和 `pytorch` 两个 skill，最终路线是：不改 MoE 数学定义、不用 DeepSpeed/分布式/容量裁剪，只优化单卡训练时的张量生命周期。第一版用纯 PyTorch 自定义 autograd 拿稳正确性和大幅降显存；第二版只在 H20 实测瓶颈明确后，再把 RMSNorm、SwiGLU 或分发聚合热点替换为 Triton。
>
> Mac mini 负责开发、接口检查、小尺寸 CPU correctness；H20 负责 CUDA bf16 正确性、8K/128K 显存速度调参和最终排名版本确认。
>
> ## Key Changes
>
> - 保留 `solution.py` 单文件提交，入口仍是 `MoEBlockOptimized`，所有参数名严格匹配 baseline，确保 `load_state_dict(strict=True)` 通过。
>
> - Router 保持原数学：`linear -> softmax(float32) -> topk -> topk 权重归一化 -> cast`。不加入 load balancing loss、capacity factor、drop token、expert choice routing，因为这些会改变赛题数学定义。
>
> - Routed MoE 去掉 baseline 的巨大 `one_hot`：把 `selected_experts [N, K]` 展平成 `[N*K]`，按 expert 排序，用 `bincount` 或 `unique_consecutive` 得到每个 expert 的 token 段，逐 expert 计算并 `index_add_` 回 `[N, H]`。
>
> - Shared expert 使用分块重计算：forward 不保存 `gate_proj(x)`、`up_proj(x)`、`SiLU(gate)*up` 的完整 128K 激活；backward 按 chunk 重算并累加输入梯度和三组 shared expert 参数梯度。
>
> - Routed experts 也使用自定义 `torch.autograd.Function`：forward 只保存 `hidden_flat`、`selected_experts`、`routing_weights`、排序索引和参数引用；backward 逐 expert 重算 `gate/up/down`，计算 `hidden`、`gate_up_proj`、`down_proj` 和 routing weight 梯度。
>
> - RMSNorm 与 MoE 合并路径统一管理：forward 生成 `combined = routed + shared` 后立刻 RMSNorm；backward 重算或分块恢复 `combined`，用 RMSNorm 公式得到 `grad_combined` 和 `post_norm.weight.grad`，再分发给 routed/shared 两路。
>
> ## Implementation Details
>
> - 默认 token chunk size 设为 `8192`，H20 上测试 `4096 / 8192 / 16384`，最终用固定启发式：`N <= 8192` 用整块，`N > 8192` 用 H20 实测最优 chunk。
>
> - 第一版不使用 `torch.compile` 作为必要路径，避免评测冷启动、图断裂和动态 shape 风险；可以保留可关闭的内部实验开关，但提交默认走稳定 eager + custom autograd。
>
> - 第一版不依赖外部 CUDA extension，避免编译环境不一致；Triton 作为第二阶段可选优化，只用于局部 kernel，并提供 PyTorch fallback。
>

  - 平均耗时 `560.27 ms -> 706.89 ms`，慢约 `26%`

- 全长度扫描里，`262144` 也能稳定跑通，优化版峰值显存约 `17.85 GB`，没有 OOM。

**为什么会有这个结果**

- 去掉了 baseline 里巨大的 `one_hot` 路由临时张量，这是最直接的显存收益点。
- routed experts 改成按 expert 分组处理，并在 backward 里重算激活，避免保存大量中间值。
- shared expert 用 chunked checkpoint，避免把 6144 宽的激活在长序列里整块留住。
- RMSNorm 也做了自定义 backward，进一步压掉反向图里的中间保存。

**结论**

这版属于“明显换显存、接受一定长序列速度损失”的方案。对比赛来说，它的优势很明确：`128K` 的显存从 33.6GB 压到 10.1GB，正确性也过了；接下来如果想继续冲排名，重点就是在不破坏正确性的前提下，把 shared expert 和 routed expert 的重算成本再往下压一点。

当前版本已经很强：128K 从 baseline `33.6GB` 降到 `10.05GB`，速度 `687ms`。下一步最有价值的方向不是再盲目写 Triton，而是做这三件事：

1. **融合 block 级生命周期，继续降显存**
   现在仍然会同时出现 `routed_output`、`shared_output`、`combined/output` 这类 `[N, H]` 大张量。128K 下每个约 `512MB bf16`。如果把 routed/shared 直接累加进一个 combined buffer，再接 RMSNorm，可以再省大约 `0.5GB-1.5GB` 峰值。这个方向更符合显存 6 分权重，值得优先做。

2. **Triton 化 routed expert 的加权写回**
   现在每个 expert 仍靠 PyTorch `index_add_` 和局部 autograd 循环。完全重写 GEMM 不划算，但可以把 `current_hidden * routing_weight + scatter/index_add` 融成更轻的 Triton kernel。收益可能是 `10-30ms`，风险中等。

3. **做双版本提交策略**
   保留当前版本作为 `stable`，再冲一个 `aggressive`。如果 aggressive 在 H20 上 128K 显存低于 `9GB` 或速度低于 `660ms` 且 correctness 全过，就用 aggressive；否则提交 stable。第一名通常靠这个策略，不靠单一路线硬赌。

我不建议继续做的方向：

- shared expert 手写 backward：我们测过，显存涨到 `12.8GB`，不划算。
- routed expert 手写 backward：只快几 ms，但显存涨，综合分不优。
- 改路由、drop token、capacity factor：违规。
- 大规模自定义 CUDA extension：时间和环境风险都比 Triton 高。

我的判断：**还有 10%-20% 的冲刺空间**。最值得做的是“融合 combined buffer + RMSNorm 生命周期”，目标把 128K 峰值从 `10.05GB` 压到 `9GB` 左右；速度目标从 `687ms` 压到 `660-670ms`。这比单纯追速度更可能提高总分。

- shared expert 的大 GEMM 反向临时 workspace；
- checkpoint 重算期间的局部激活和 GEMM workspace；
- 参数梯度张量本身；
- routed expert 分组过程和 autograd 保存的必要状态；
- PyTorch/CUDA allocator 的峰值保留行为。

也就是说，继续压显存不能靠“少建一个变量”解决了，要做更深的东西，比如：

- 自定义 shared expert backward，分块手写梯度，但之前试过一版，显存反而涨，需要更精细实现。
- Triton/CUDA fused SwiGLU + down projection 的局部 kernel，但 GEMM 仍然是核心，难度高。
- 把 shared expert backward 拆得更细，控制每次 GEMM 的 workspace 和中间生命周期。
- 写更完整的 MoE core 自定义 autograd，让 routed/shared/RMSNorm 的 backward 统一调度。

短期判断：当前版本已经是**高性价比稳定版**。如果目标是稳进高分，应该提交当前版本；如果目标是赌第一，还可以继续做 aggressive 深水区优化，但每多拿一点收益都要用更多 H20 时间和正确性风险去换。

1. **自定义 Router backward**
   - forward 仍然完全保持 `linear -> softmax(fp32) -> topk -> normalize`。
   - backward 分块重算 `gate_logits/softmax/topk`，手写 softmax + topk normalize 的梯度。
   - 好处是进一步减少 PyTorch 为 router 保存的中间图和临时张量。
   - 风险是梯度细节更容易踩数值边界，但只要通过 correctness，就是合规大招。

2. **按序列长度动态 chunk 策略**
   - 现在默认偏显存：`32768/32768`。
   - 可以在 `solution.py` 里根据 `N` 自动选择：
     - `N <= 8192` 用速度更优路径。
     - `N >= 131072` 用显存最优路径。
   - 这个不改数学定义，只是调执行策略，适合评测覆盖多长度时偷分。

3. **保存索引压缩**
   - `selected_experts/top_pos/sorted_experts` 这些索引目前大概率有不少 `int64`。
   - expert id 只有 128，top pos 只有 8，理论上可以保存成 `uint8/int16`，backward 用时再转回 `int64`。
   - 单项收益不会像 chunk 那么夸张，但可能白捡几十 MB 到一两百 MB，风险较低。

4. **最终版打两个 profile 靶点**
   - 一个是“极限显存版”：目标 128K 冲 `6.5GB` 附近。
   - 一个是“均衡排名版”：目标 128K `7GB / <690ms`。
