# 公考题库 Agent 功能需求文档

## 1. 产品定位

公考题库 Agent 第一版包含两个互补能力：

- **学习教练 Agent**：基于用户练习记录、错题和知识点统计，诊断薄弱点，并推荐下一组训练。
- **讲题助教 Agent**：围绕单道题提供可追问的解析、做题思路、易错点和同类题迁移提示。

第一版不做开放式聊天入口。Agent 必须贴着刷题闭环工作：练完后知道下一步练什么，卡题时知道这题为什么错。

## 2. 目标用户与核心场景

目标用户：

- 高频刷题用户：需要减少盲目刷题，提高每次练习收益。
- 错题较多用户：需要知道错题背后的知识点问题，而不是只看答案。
- 移动端用户：需要在碎片时间里快速得到下一步行动。

核心场景：

- 用户提交一场练习后，系统给出“本次诊断”和“下一组推荐训练”。
- 用户在结果页或错题页点开某题，向讲题助教追问为什么选错、如何快速判断、同类题怎么做。
- 用户进入首页或练习记录页时，学习教练根据最近表现给出今日训练入口。

## 3. 非目标

第一版不做：

- 长周期备考日历和每日任务排期。
- 无上下文的通用 AI 问答。
- 直接替代官方解析的自动判题依据。
- 生成新题或改写题库原题。
- 高风险承诺，例如“保证提分”“预测真题”。

## 4. 学习教练 Agent

### 4.1 目标

学习教练 Agent 负责回答一个问题：用户下一步最该练什么。

### 4.2 触发位置

- 练习提交结果页：立即展示本次诊断和推荐训练。
- 题库首页 / 我的题库：展示今日推荐。
- 练习记录页：展示近 7 天或近 20 场练习的薄弱点。
- 错题本页：推荐优先复习的错题分类。

### 4.3 输入数据

可使用现有数据：

- `PracticeSession`：练习模式、题数、正确率、用时、提交时间。
- `PracticeAnswer`：单题对错、作答、耗时、题目顺序。
- `Question`：题型、难度、知识点、全站正确率、来源。
- `QuestionTag`：知识点层级。
- `WrongQuestion`：错题次数、最近错题时间、是否已解决。
- `UserTagStats`：知识点维度答题数、正确数、错误数、正确率。

第一版推荐避免依赖外部长期记忆，优先用数据库内可解释数据。

### 4.4 输出内容

学习教练 Agent 输出一张结构化卡片：

- 诊断标题：一句话说明当前最大问题。
- 证据：列出 2-3 条来自练习数据的依据。
- 推荐训练：给出 1-3 个可点击动作。
- 预期收益：说明为什么现在练这个。
- 置信度：高 / 中 / 低。

示例：

```json
{
  "title": "资料分析正确率低且耗时偏长",
  "evidence": [
    "近 5 次练习中资料分析正确率 48%",
    "该模块平均每题用时 93 秒，高于个人平均 41 秒",
    "未解决错题中资料分析占 12 道"
  ],
  "recommendations": [
    {
      "type": "SPECIAL_PRACTICE",
      "label": "练 10 道资料分析中等题",
      "tagId": "tag_id",
      "difficulty": "MEDIUM",
      "count": 10
    },
    {
      "type": "WRONG_MEMORIZE",
      "label": "背 10 道资料分析错题",
      "tagId": "tag_id",
      "count": 10
    }
  ],
  "confidence": "HIGH"
}
```

### 4.5 推荐规则

第一版可以先使用规则引擎，LLM 只负责把诊断翻译成自然语言：

- 优先推荐低正确率且答题数足够的知识点。
- 如果某知识点错题未解决数量高，优先推荐错题练习或背题模式。
- 如果正确率尚可但单题耗时明显高，推荐限时专项练习。
- 如果用户最近练习量少，推荐每日一练。
- 如果数据不足，推荐基础专项练习，而不是给出强诊断。

### 4.6 推荐动作

第一版支持以下动作：

- 开始专项练习：调用现有专项组题能力。
- 开始错题练习：调用现有错题练习能力。
- 开始背题模式：调用现有 `MEMORIZE` 模式。
- 查看诊断详情：进入练习记录或知识点统计。

## 5. 讲题助教 Agent

### 5.1 目标

讲题助教 Agent 负责回答一个问题：这道题为什么这样做，以及用户为什么会错。

### 5.2 触发位置

- 练习提交后，每道错题旁显示“问助教”。
- 背题模式中，每道题显示“讲解”入口。
- 错题详情页支持追问。

普通未提交练习中默认不开放完整讲解，避免提前泄露答案；如后续要支持“提示模式”，应单独设计。

### 5.3 输入数据

- 当前题题干、材料、选项。
- 正确答案和官方解析。
- 用户答案。
- 题目知识点、难度、来源、全站正确率。
- 用户在该题上的历史错误次数。

### 5.4 输出内容

讲题助教回答应包含：

- 结论：正确答案是什么。
- 关键思路：本题最短解题路径。
- 错因分析：用户答案为什么不对。
- 易错点：同类题常见陷阱。
- 迁移提示：下次遇到类似题先看什么。

### 5.5 交互问题

提供快捷追问：

- “为什么不选我选的这个？”
- “有没有更快的做法？”
- “这题考哪个知识点？”
- “给我总结成一句口诀”
- “下次怎么识别同类题？”

### 5.6 安全与边界

- 助教不得修改题库标准答案。
- 如果官方解析为空，必须明确说明“当前题缺少官方解析”，再基于题干做辅助分析。
- 如果题干或选项含图片且缺少可读文本，必须提示解析可能不完整。
- 回答应避免输出与公考无关的泛聊天内容。

## 6. 用户体验

### 6.1 结果页体验

提交后展示顺序：

1. 本次练习结果。
2. 学习教练诊断卡片。
3. 推荐训练按钮。
4. 错题列表和讲题助教入口。

### 6.2 首页体验

首页展示一个轻量入口：

- 有足够数据：显示“今日建议练习”。
- 数据不足：显示“先完成一次每日一练，Agent 才能给出更准建议”。

### 6.3 单题体验

讲题助教以抽屉或底部面板出现，保持题目上下文可见。移动端优先使用底部抽屉，桌面端可使用右侧面板。

## 7. 接口需求

### 7.1 获取学习诊断

```txt
GET /api/agent/coach/diagnosis?scope=recent
```

返回：

```json
{
  "ok": true,
  "data": {
    "summary": {},
    "recommendations": []
  },
  "error": null
}
```

### 7.2 获取单场练习诊断

```txt
GET /api/agent/coach/sessions/{sessionId}
```

用途：结果页展示本次诊断。

### 7.3 创建推荐训练

```txt
POST /api/agent/coach/recommendations/{recommendationId}/start
```

用途：将推荐动作转成实际 `PracticeSession`。

### 7.4 单题讲解

```txt
POST /api/agent/tutor/questions/{questionId}
```

请求：

```json
{
  "sessionId": "session_id",
  "prompt": "为什么不选 B？"
}
```

返回：

```json
{
  "ok": true,
  "data": {
    "answer": "讲解内容",
    "suggestedPrompts": []
  },
  "error": null
}
```

## 8. 数据需求

第一版可不新增核心刷题表，但建议新增 Agent 相关表以便追踪效果：

- `AgentRecommendation`：记录推荐内容、来源、置信度、是否被点击。
- `AgentTutorMessage`：记录单题追问上下文，便于继续对话和质量评估。
- `AgentFeedback`：记录用户对诊断或讲解的反馈。

是否把 Agent 输出持久化，是后续需要单独拍板的设计决策。

## 9. 技术架构建议

第一版 Agent 采用“Next.js 单体内置 Agent 服务层 + LangGraph.js 编排”的架构，不单独拆 Python Agent 服务。

推荐技术栈：

- **LangGraph.js**：作为 Agent 核心编排框架，承载状态图、工具节点、条件分支、checkpoint 和 human-in-the-loop 扩展点。
- **LangSmith**：作为 Agent tracing、debug、数据集回放和评测平台。
- **OpenAI Responses API**：作为默认模型调用接口，支持工具调用和结构化输出。
- **Vercel AI SDK**：用于前端讲题助教的 streaming UI 和消息流适配，不承载核心业务决策。
- **Zod**：定义 Agent 输入、输出、工具参数和结构化结果 schema。
- **Prisma / MySQL**：继续作为用户、练习、错题、推荐记录的业务数据源。
- **Qdrant 或其他向量库**：后续用于相似题检索、错因案例检索和知识点 RAG；第一版不是必须依赖。

推荐目录：

```txt
src/server/agent/
  graph/
    coach-graph.ts
    tutor-graph.ts
    state.ts
  tools/
    get-user-practice-stats.ts
    get-question-context.ts
    create-practice-session.ts
    search-similar-questions.ts
  prompts/
    coach.ts
    tutor.ts
  schemas/
    coach-output.ts
    tutor-output.ts
  evals/
    coach-cases.test.ts
    tutor-cases.test.ts
```

学习教练 Agent 使用规则推荐引擎主导，LangGraph.js 编排数据获取、薄弱点分析、推荐动作生成和自然语言解释。讲题助教 Agent 使用 LangGraph.js 编排题目上下文获取、讲解生成、追问建议和安全边界检查。

## 10. 成功指标

学习教练 Agent：

- 推荐训练点击率。
- 推荐训练完成率。
- 用户 7 日留存。
- 被推荐知识点后续正确率变化。

讲题助教 Agent：

- 单题讲解打开率。
- 追问次数。
- 用户反馈“有帮助”比例。
- 错题后续解决率。

## 11. MVP 范围

第一阶段：

- 结果页展示学习教练诊断卡片。
- 支持 1-3 个推荐训练动作。
- 错题旁提供讲题助教入口。
- 讲题助教支持快捷追问。

第二阶段：

- 首页今日推荐。
- 知识点趋势诊断。
- 讲题对话历史。
- 用户反馈闭环。

第三阶段：

- 长周期备考规划。
- 个性化复习节奏。
- 多轮学习计划执行追踪。

## 12. 待决策问题

- 第一版讲题助教是否允许在未提交时提供“提示但不泄露答案”？
- Agent 输出是否需要持久化到数据库？
- 学习教练诊断周期采用近 7 天、近 20 场，还是二者结合？
- 推荐训练是否允许自动创建练习会话，还是必须用户确认后创建？
- LangGraph checkpoint 第一版是否持久化到数据库，还是仅保留 LangSmith trace 和业务结果？
